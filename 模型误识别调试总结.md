更新时间：2026-03-19

---

## 1. 问题定义

当前项目的核心问题不是“检不出排球”，而是 **误识别 / Query Over-Activation（OA）**：

- 有排球时，D-FINE 基本能正确检出
- 无排球时，会把人、球网、地板、反光、边缘物体等高置信度误识别为排球
- 这类问题在 AGX Orin 部署时最明显

该问题不是普通阈值问题，而是 D-FINE / DETR 类模型在 **单类微调** 场景下的典型问题。

相关背景分析见：`query-over-activation.md`

---

## 2. 为什么 D-FINE 容易出现这种误识别

D-FINE 是 **query-based 的 DETR 系列模型**，不是 YOLO 这种 dense detector。

关键事实：

- 默认 `num_queries=300`：`D-FINE/src/zoo/dfine/dfine_decoder.py:467`
- 这些 learned queries 在 COCO / Objects365 预训练中学会了“哪里有物体”
- 单类微调后，模型很容易把“这里有物体”直接等价成“这里有排球”

这就是当前误识别的结构性来源。

因此，当前模型调试优先级应围绕：

1. 减少 query 过激活槽位
2. 增强背景 / no-object 训练信号
3. 用 OA 结果辅助选模
4. 避免为了 AP 小幅提升而引入过高算力成本

**结论：当前阶段不应优先增加分辨率。** 这会明显增加算力消耗，但不是解决误识别的主路径。

## 3. 已尝试过的方法总览

### 3.1 方案 A：负样本图并入训练（`train_with_negatives.json`）

这条路已经明确尝试过，不是未做。

相关文件：

- `负样本微调/prepare_hard_negatives.py`
- `config/volleyball_s_finetune_negatives.yml`
- `config/volleyball_s_obj2coco_neg_d.yml`
- `config/volleyball_s_obj2coco_neg_d_bg2.yml`

实现方式：

- 将 `coco/images/negative_samples/` 中的误检图像作为**空标注图片**并入训练集
- 生成：`coco/converted/annotations/train_with_negatives.json`

生成脚本路径：
- `负样本微调/prepare_hard_negatives.py`

该脚本会：
- 读取 `train.json`
- 扫描 `negative_samples/`
- 将负样本图追加到 `images`
- 保持 `annotations` 为空
- 输出 `train_with_negatives.json`

### 3.2 方案 D：DN 随机背景 queries

已实现到训练代码里。

关键代码：
- `D-FINE/src/zoo/dfine/denoising.py:85-144`

机制：
- 每个 batch 注入随机背景框
- 与 GT 重叠的随机框会被过滤
- 随机背景 query 被标记为背景类参与分类训练

已实现参数：
- `num_neg_random`
- `neg_iou_threshold`

### 3.3 背景分类权重增强（`bg_loss_weight`）

已实现到损失函数。

关键代码：
- `D-FINE/src/zoo/dfine/dfine_criterion.py:114-117`

机制：
- 对未匹配 query 的分类损失额外加权
- 强化 no-object / 背景抑制

### 3.4 OA 训练中评估与 `best_oa.pth`

已实现。

关键代码：
- OA 后台线程启动：`D-FINE/src/solver/det_solver.py:126-148`
- `best_oa.pth` 保存逻辑：`D-FINE/src/solver/det_solver.py:206-214`

当前逻辑：
- 训练中可评估 OA
- 当 AP 达到 `oa_ap_min` 后，OA 结果才允许进入 `best_oa.pth` 保存决策

**重要限制：**
当前 `oa_ap_min` 只控制“是否保存 `best_oa.pth`”，**不控制是否启动 OA 评估线程**。
也就是说，当前代码并没有真正实现“AP 低于 0.84 时不跑 OA 评估”。

### 3.5 q100（减少 queries）

历史上已经尝试过，但曾经与 freeze backbone 绑定在一起。

相关配置：
- `config/volleyball_s_obj2coco_neg_d_q100_freeze.yml`

关键参数：
- `num_queries: 100`
- `num_top_queries: 100`
- `freeze_at: 3`
- `freeze_norm: True`

这条路的方向是对的，但历史版本把：
- `q100`
- `freeze backbone`

绑在一起了，所以很难单独判断 `q100` 的净收益。

### 3.6 top-k 越界问题

历史上已经出现过相关 bug，并修过。

典型报错：
- `RuntimeError: selected index k out of range`

根因：
- 当 `num_queries` 从 300 降到 100 时，如果 `num_top_queries` 仍为 300，后处理会在 `topk` 时报错

相关代码：
- `D-FINE/src/zoo/dfine/postprocessor.py:46-54`

因此，只要用 q100，就必须同步设置：

```yaml
DFINEPostProcessor:
  num_top_queries: 100
```

## 5. 当前最重要的结论

### 5.1 A 已经试过，而且不应再作为当前主线

原因不是 A 完全没用，而是：
- A 的 OA 指标可能会更好看
- 但真实部署效果不一定稳定
- 你无法收集所有负样本类型
- 把“误检治理”过度绑定到已收集负样本，会造成泛化受限

因此，当前方向应转向：
- **no-A**
- 更少 queries
- 更强背景损失
- AP 达标后再做 OA 评估与选模

### 5.2 当前不应优先提分辨率

虽然提高分辨率可能提升 `AP_s`，但它：
- 算力成本高
- 不是误识别问题的主杠杆
- 对 D-FINE 这种 query-based 模型，OA 的根子主要在 query 容量和背景分类，而不是像素分辨率

### 5.3 q100 是当前最该优先验证的方向

原因：
- 直接对准 D-FINE 的 query 过激活根因
- 同时降低计算量
- 比加分辨率更符合当前目标
- 历史上虽然试过 q100，但和 freeze 绑在一起，不够干净

因此当前应验证的是：

> **q100，但不要 freeze，不要 A，保留 D 和 bg2**

## 7. 当前 OA 逻辑存在的问题

理想需求是：

- AP < `oa_ap_min` 时，不评估 OA
- AP >= `oa_ap_min` 后，才启动 OA 评估
- 并继续按 OA 保存 `best_oa.pth`

但当前代码不是这样。

当前代码行为：
- `eval_overactivation=True` 时，每个 epoch 都会启动 OA 线程
- `oa_ap_min` 只控制 `best_oa.pth` 是否允许保存

关键代码：
- `D-FINE/src/solver/det_solver.py:126-148`
- `D-FINE/src/solver/det_solver.py:206-214`

所以当前正确的长期做法应是：

### 需要代码修改
把 `det_solver.py` 改成：
- 只有当当前 AP >= `oa_ap_min` 时才真正启动 OA 线程

这会实现你真正要的语义：
- AP 不够时不浪费评估时间
- AP 达标后再按 OA 选模型

## 11. 最终结论

当前关于“模型误识别”的最重要判断如下：

1. **A 方案已经试过，不应再作为当前主线**
2. **当前目标不是提分辨率，而是压制 Query Over-Activation**
3. **D-FINE 是 query-based 模型，q100 是当前最值得优先验证的方向**
4. **当前推荐主线是：no-A + q100 + D + bg2，不 freeze**
5. **当前 `oa_ap_min` 只影响保存，不影响是否评估；后续应改代码实现 AP 门控 OA**

如果只保留一句话：

> 现在最该做的，不是继续收集负样本，也不是提高分辨率，而是用 no-A 路线验证 q100 是否能从结构上减少 D-FINE 的 query 误激活，并在 AP 达标后再启用 OA 评估与选模。
