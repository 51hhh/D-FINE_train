# D-FINE 排球单目标训练总结

更新时间：2026-03-12

## 1. 项目当前结论

当前仓库的主训练路线已经比较明确：

- **主力方案**：`S` 模型 + `obj+coco` 预训练迁移微调
- **不推荐主用**：仅 `obj` 预训练
- **不作为当前主线**：`N` 模型（主要原因是没有 `obj+coco` 预训练权重）
- **从零训练可行，但收敛更慢**：`exp_s_transfer_aug_2000`

当前经验结论：

1. `obj+coco` 预训练效果明显优于仅 `obj` 预训练
2. 不做类别绑定、直接丢弃检测头重新适配自定义单类任务，是当前有效方案
3. 对单目标排球任务，当前已经不是“能否收敛”的问题，而是“如何继续提升小目标定位质量”的问题

## 2. 数据集与标注结论

数据集未直接放入仓库，但根据 `benchmark/scripts/prepare_dataset.py` 的逻辑，可以确认当前数据转换方式是正确的。

### 2.1 数据来源

- 输入：已标注 XML + 图片
- 输出：COCO 格式标注文件（`train.json` / `val.json` / `test.json`）

### 2.2 类别 ID 结论

`prepare_dataset.py` 中类别映射逻辑：

```python
cls2id = {name: i for i, name in enumerate(sorted(cls_names))}
```

这说明：

- 类别 ID 从 `0` 开始
- 对当前单类 `volleyball` 任务，`category_id=0` 是正确的
- 因此：
  - `num_classes: 1` 正确
  - `remap_mscoco_category: False` 正确

### 2.3 当前推荐数据路径写法

在 `D-FINE` 目录下启动训练时，推荐统一写成相对路径：

```yaml
img_folder: ../coco/images
ann_file: ../coco/converted/annotations/train.json
```

验证集和负样本微调同理：

```yaml
img_folder: ../coco/images
ann_file: ../coco/converted/annotations/val.json
ann_file: ../coco/converted/annotations/train_with_negatives.json
```

## 3. 当前配置修正结果

本轮已经完成以下配置修正：

### 3.1 路径统一为相对路径

已统一以下配置文件的数据路径：

- `config/volleyball_n_transfer_fast.yml`
- `config/volleyball_s_finetune_negatives.yml`

`config/volleyball_s_transfer.yml` 原本就是相对路径，无需修改。

### 3.2 单卡训练关闭 SyncBN

针对当前训练环境：

- 云服务器单卡 `4070S`
- Linux 单卡 `4090D`

当前统一建议：

```yaml
sync_bn: False
```

原因：

- `SyncBN` 主要用于多卡 DDP
- 单卡训练通常没有收益
- 还会增加额外复杂度

### 3.3 补齐 N 配置验证集变换

已补充 `config/volleyball_n_transfer_fast.yml` 验证集中的：

```yaml
- {type: ConvertBoxes, fmt: 'cxcywh', normalize: True}
```

这样训练集/验证集的数据处理链更一致。

## 4. 当前各配置定位

### 4.1 `volleyball_s_transfer.yml`

用途：

- 当前主训练配置
- `S` 模型 + `obj+coco` 预训练迁移

特点：

- 单类检测：`num_classes: 1`
- 输入尺寸：`640`
- 总轮次：`160`
- 多尺度训练前期启用，后期关闭
- 当前最推荐作为主实验配置

结论：

- **这是当前最优主线配置**
- 后续优化建议主要围绕它继续展开

### 4.2 `volleyball_s_finetune_negatives.yml`

用途：

- 困难负样本微调
- 针对误检进行进一步压制

特点：

- 使用 `train_with_negatives.json`
- 学习率更低
- 微调轮次较短
- 适合在主模型已经较强时做第二阶段精修

结论：

- **方向正确，适合做二阶段优化**
- 但负样本比例需要控制，不能无限增加

### 4.3 `volleyball_n_transfer_fast.yml`

用途：

- 边缘部署导向的轻量对照实验
- 更强调速度和吞吐

特点：

- 输入尺寸：`512`
- 模型更小
- 当前不是主实验线

结论：

- **适合作为速度基线/对照组**
- 不适合作为当前精度主线

### 5.1 关键结论

可重点关注：

- `AP50:95 = 0.844`
- `AP50 = 0.994`
- `AP75 = 0.969`
- `AP_s = 0.751`

解读：

1. **AP50 已经非常高**
   - 说明模型“能不能检测到球”这件事已经基本解决
2. **AP75 也很高**
   - 说明定位质量已经不错
3. **AP_s 仍有优化空间**
   - 对排球这种小目标任务，后续主要提升空间在小目标高质量定位
4. **当前阶段更像后期精修，不是前期找配置**

### 5.2 关于 loss 的判断

当前 `train_loss` 绝对值较大，但不应直接按常规分类/检测任务直觉解读。

原因：

- D-FINE 含有多组损失项
- 包括：`aux / pre / enc / dn / bbox / giou / fgl / vfl / ddf`
- 更重要的是看验证指标，而不是单独看 `train_loss` 的数值大小

当前验证指标已经表明模型有效，说明训练是正常的。

## 6. 模型选择结论

### 当前结论

- **最佳主线**：`S + obj+coco`
- **次优或不推荐主用**：`obj-only`
- **从零训练**：可做对照，但收敛时间更长
- **N 模型**：当前不作为主线，主要因为缺少合适预训练权重

### 原因总结

1. `obj+coco` 提供了更强的目标检测先验
2. 对单目标排球任务，小目标泛化能力更重要
3. 当前实测已经证明 `obj+coco` 收敛更快、上限更高

---

## 7. 后续优化优先级（强烈建议按顺序做）

### 7.1 第一优先：提高输入分辨率

当前配置为 `640`，后续最值得做的实验：

- `640 -> 704`
- `640 -> 768`

预期：

- `AP50` 提升可能不大
- `AP75` 与 `AP_s` 更可能提升
- 对排球这种小目标任务，收益通常比改 loss 更直接

风险：

- 显存占用提升
- 吞吐下降

### 7.2 第二优先：调整多尺度训练停止时机

当前主配置：

- `stop_epoch: 56`

建议做对照实验：

- `56`
- `80`
- `100`

目的：

- 找到更适合排球任务的前期泛化 / 后期收敛平衡点

### 7.3 第三优先：系统化困难负样本微调

建议测试困难负样本比例：

- 正样本图数的 `10%`
- 正样本图数的 `20%`
- 正样本图数的 `30%`

经验判断：

- 太少：抑制误检效果不明显
- 太多：模型可能变得过于保守，召回下降

### 7.4 第四优先：检查标注精度

当模型已经达到当前指标水平后，标注质量对 `AP50:95` 和 `AP75` 的影响会非常明显。

建议重点抽查：

- 远距离小球
- 运动模糊球
- 半遮挡球
- 图像边缘球
- 易与灯光反光点混淆的误检样本

### 7.5 第五优先：多 seed 对照

建议至少做：

- `seed=0`
- `seed=1`
- `seed=2`

原因：

- 避免被单次偶然最优结果误导
- 更适合判断一个配置是否真的更优

## 8. 推荐实验表

建议按下面顺序做实验：

| 实验编号 | 目标 | 修改项 | 预期收益 | 风险 |
|---|---|---|---|---|
| E1 | 提升小目标定位 | 输入 `640 -> 704` | `AP75/AP_s` 可能提升 | 显存增加 |
| E2 | 继续提高上限 | 输入 `640 -> 768` | 小目标最有希望继续涨分 | 速度下降明显 |
| E3 | 优化前后期训练平衡 | `stop_epoch=56 -> 80` | 后期定位可能更稳 | 训练周期变长 |
| E4 | 继续观察收敛上限 | `epochs=160 -> 200` | 可能带来后期小幅提升 | 可能收益有限 |
| E5 | 压制误检 | 负样本比例 `10/20/30%` | FP 降低 | 召回可能下降 |
| E6 | 验证稳定性 | 多 `seed` 对比 | 判断是否真实提升 | 训练成本增加 |

## 9. 当前不建议优先做的事

以下方向暂时不建议优先投入：

1. **频繁改 loss 权重**
   - 当前主要瓶颈更像数据与分辨率，不像损失设计
2. **频繁换优化器**
   - 当前优化器和基础训练流程已经有效
3. **过早切换到 N 模型主训**
   - 当前最强主线仍然是 `S + obj+coco`
4. **盲目增加负样本数量**
   - 容易导致模型过保守

---

## 10. 最终建议（简版）

如果只保留最重要的结论，当前建议如下：

1. 继续以 **`volleyball_s_transfer.yml`** 作为主训练配置
2. 保持当前单卡环境下 **`sync_bn: False`**
3. 优先做 **更高输入分辨率实验（704 / 768）**
4. 再做 **多尺度停止轮次** 对照实验
5. 最后再做 **困难负样本比例实验**
6. 后期想继续抠分时，优先检查 **小球标注框质量**

---

## 11. 一句话总结

当前项目已经从“配置是否正确”阶段，进入到“围绕 `S + obj+coco` 主线继续打磨小目标定位质量”的阶段。

后续最值得投入的方向，不是大改模型结构，而是：

- 更高分辨率
- 更合理的训练阶段划分
- 更干净的困难负样本
- 更精细的小球标注质量
